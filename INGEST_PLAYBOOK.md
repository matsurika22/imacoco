# 報告投入 運用書 (Claude Code 用)

このファイルは「黒服から受け取った LINE 報告のテキストを Claude が読み取り、
`ingest.py` で DB に流し込むまで」を**他セッションでもブレずに**やるための運用書。
仕様の単一情報源は `CLAUDE.md`。本書はそれを補う実務マニュアル。

新しい会話を始めるたびに本書と `CLAUDE.md` を通読すること。

---

## 0. 大方針 (絶対に守る)

ユーザー(コンサル)は技術用語を覚えていない。次の3つを守る。

1. **カラム名・英語 enum・テーブル名をユーザーに見せない**
  内部表現は会話の外に出さない。日本語の意味で話す。具体例は §10。
2. **判定基準もユーザーは覚えていない前提で聞く**
  「これで done でいいですか?」のように **enum 値の選択肢を聞かない**。
  「これは『動画もらえてゴール達成』扱いにしますか? それともまだ追跡を続けますか?」
  のように **意味そのもの**を聞く。
3. **基本的に全部聞く(独断禁止)**
  reaction / ステータスの変化 / 日付の解釈 / 退店時期 / 理由の文言 /
  履歴メモの文言 / 現状サマリの文言 — 迷わなくても聞く。

> なぜ: ユーザーはコンサル業務側で、DB スキーマや施策ごとの完了条件の細目を
> 覚えていない。Claude が技術用語で確認すると詰まって作業が止まる。
> 意味の言葉で問えば即答できる。

## 1. 入力フォーマット

ユーザーは基本 **1メッセージ = 1報告(もしくは数件まとめて)** をチャットに貼り付ける。

### 1-1. 退店リスク報告 (確認済みサンプル)

```
のん 5月前半

家庭の事情

あすか

時期は未定

痩せたので錦の夜に行きたい
```

- **空行で区切られたブロックが意味の単位**
- 1人につき複数ブロック(キャスト名+時期 / 理由)に分かれることがある
- 報告主の黒服名は本文に書かれない場合がある → **キャストの担当黒服から自動で引く**(§4)

### 1-2. A群施策(TikTok / 紹介CP / バースデー)の報告

**典型フォーマット**: 「日付 → キャスト名 → 各施策に 1〜数行のコメント」が縦に並ぶ。
1メッセージに複数キャストが入る。例:

```
5/3
みく
もしかしたら友達紹介してくれそう
TikTok動画買取は前向きに検討

あすか
紹介は出来なそう(友達がいない)
TikTok動画買取は前向きに検討
```

施策キーワードのマッピング:
- 「友達紹介」「紹介」 → 紹介CP (initiative 2)
- 「TikTok」「TikTok動画(買取)」 → TikTok動画買取 (initiative 1)
- 「バースデー」「誕生日イベント」 → バースデー (initiative 3)
- 「やめる」「退店」「他店に行きたい」「◯月まで」など → 退店リスク (`add-quit-risk`)

#### 1-2-α. 判定パターン早見表

**※ ここに無い言い回し / 微妙な判定が出てきたら必ず個別確認。合意できたら即本表に追記する(§13)。
※ ここにある言い回しと一致する報告が複数件まとまって来た時は「これ全部パターン推論でいい?」と
一括確認すれば、それ以降同類はノー質問で投入してよい。**

##### 紹介CP (施策2) — declined (無理だった)

| 報告本文の典型表現 | reaction | content/comment 例 |
|---|---|---|
| 「友達紹介は期待薄い」「期待薄」 | negative | 「期待薄いとのこと」 |
| 「友達がいない/少ない/少なすぎて」「友達紹介は出来なそう」 | negative | 「友達いないので期待薄」「友達少なすぎて難しい」 |
| 「みんな働いてる/岡崎だから難しい」 | negative | 「みんな働いてる&岡崎で難しい」 |
| 「黒服側ならできるかも、でも(キャスト紹介は)期待薄」 | negative | 「期待薄(黒服側ならとのこと)」 |

##### 紹介CP (施策2) — in_progress (進行中)

| 報告本文の典型表現 | reaction | content/comment 例 |
|---|---|---|
| 「もしかしたら紹介してくれそう」 | positive | 「友達紹介してくれそう」 |
| 「探してみる」「何人かに声かけてみる」 | positive | 「探してみるとのこと」「何人かに声かけてみる」 |
| 「元体験来た友達に再アプローチ」(過去不採用者へ再打診) | positive | 「元体験友達に再アプローチ予定」 |
| 「今回は他店に行ってしまったが、また機会あるかも」(他店入店話) | neutral | 「今回は他店へ、以降追跡」 |
| 「(条件あるが)体験呼んでみる」 | positive | 「ぽっちゃり/週2-3だが体験呼んでみる」(呼ぶだけはまだ done ではない) |

##### TikTok動画買取 (施策1) — declined (無理だった)

| 報告本文の典型表現 | reaction | content/comment 例 |
|---|---|---|
| 「顔出ししてない」「顔出しNG」(単独 or 動画未経験と組合せ) | negative | 「顔出しNG、動画未経験」「顔出しNG、反応イマイチ」 |
| 「顔出ししてないのでむずかしそう」「微妙」 | negative | 「顔出しNG、むずかしそう」「顔出しNG、微妙」 |
| 「出たくない」 | negative | 「出たくないとのこと」 |
| 「ブスで叩かれるからイヤ」 | negative | 「ブスで叩かれるのがイヤ」 |

##### TikTok動画買取 (施策1) — in_progress (進行中)

| 報告本文の典型表現 | reaction | content/comment 例 |
|---|---|---|
| 「前向きに検討」「前向きに検討中」 | positive | 「前向きに検討中」 |
| 「顔出しなしなら(少し)考える」(条件付き検討) | neutral | 「顔出しなしなら考える」 |
| 「叩かれそうで不安だが前向き、うまく撮れたら」(両価) | positive | 「叩かれ不安あるが前向き」 |
| 「動画未経験だが買取は前向きに検討」 | positive | 「動画未経験だが買取は前向きに検討」 |

##### バースデー (施策3)

サンプル未収集。出てきた時点で「これは典型?」とユーザー確認 → 本表に追記。
**重要**: 「やらない」と言われても declined ではなく in_progress 据え置き(CLAUDE.md §6 / §7-3)。

##### 退店リスク (施策5)

`add-quit-risk` で投入(§6 と §8 参照)。常に `--certainty confirmed`。時期未定なら `--date` 省略。

#### 1-2-β. レイアウト誤読の罠

実際に踏んだ誤読パターン。気をつける:

- **改行漏れで 2人分が 1行に詰まる**
  例: 「...前向きに検討するとのこと ゆい」 ← 「ゆい」が次キャストの名前
- **サブヘッダーが次キャストの前置きとして書かれる**
  例: 「ゆうか / ... / TikTokは前向きに検討    退店予定」 ← 「退店予定」は実は次のあすかの話の前置き
- **キャスト名が未登録**だと別人候補を疑う前にユーザーに確認(担当黒服とシフトを聞く)
- どこからどこまでが1人の話か曖昧な箇所は **パースを止めてユーザーに聞く**

#### 1-2-γ. パターン推論モードの運用

1メッセージに同類報告が複数並んだ時の流れ:
1. 早見表に一致する報告と、特殊判定が必要な報告に振り分ける
2. 特殊判定だけ個別確認質問
3. 残りは「これらパターン推論でいい?」と一括確認
4. 合意取れたらノー質問で順次投入

これにより 6 件 / 8 件のような大量バッチでも 1〜2 個の質問だけで投入完了できる。

## 2. パース手順 (1メッセージで Claude がやる流れ)

### Step 1. 仮読み + カテゴリ切り分け

メッセージ全体を読んで、次を仮置きする:

- **誰の話か**(キャスト)
- **何の話か** → 以下のカテゴリ判定:
  - 退店の話(辞める / 辞めそう / 他店に移る等) → **`add-quit-risk`** で投入
  - TikTok の話 / 紹介CP の話 / バースデーの話 → **`add-report`** で投入(A群施策、category='cast')
  - ひろし顔きき計画(施策4) → **投入不可**。Claude は「これは記録対象外の施策です」と返す(CLAUDE.md §6, §7)
- **報告日**が本文に書かれているか確認(下記 Step 1-α)

### Step 1-α. 報告日の取り扱い(運用ルール)

- 基本、ユーザーが貼り付けるメッセージに **「今日のぶん」「昨日来た」「5/8のぶん」**
  のように報告日を併記してくれる前提
- **報告日が書かれていなかったら必ず聞く**。Claude が「今日」と決めつけない
  - 聞き方: 「この報告いつ来たぶんですか? 今日(2026-MM-DD)で登録していいですか?
    それとも別の日?」
- 複数キャストの報告がまとめて貼られていて全部同じ日なら、まとめて確認 1回で OK
- 日付が確定したら、当該メッセージ内の全 report に共通で使う

### Step 2. 仮読みを日本語で確認

> 悪い例: 「のん の certainty=likely、initiative_id=5 で登録?」
> 良い例: 「のんさんが家庭の事情で 5月前半に辞める、という話で読み取りました。
>        退店の話として登録していいですか?」

### Step 3. 個別の確定情報を意味の言葉で順に聞く
聞くべき項目は §3。**1質問ずつ短く**。

### Step 4. マスタ未登録があれば登録判断を聞く
- 新キャスト → 担当黒服とシフト(夜/昼)を聞いた上で `cast add`
- 新黒服 → §11 参照(`kurofuku add` CLI が現状無いので運用判断が必要)
- **自動仮登録は絶対しない**(CLAUDE.md §9)

### Step 5. ステータス変化を伴う場合は事前確認
- ステータスが変わる/巻き戻る可能性があるときは **`show-status` で現状確認**してから提示
  例: `show-status --cast えま --initiative 1` → done だったら巻き戻しに該当
- 巻き戻し検知 → ユーザーに「これ前に『完了/無理』扱いの件です。取り消しますか?」と必ず確認(§7-2)

### Step 6. 確定したらコマンドを提示
- コードブロックで `ingest.py` 一行を見せる
- **実行可否を最後にもう一度確認**してから走らせる
- 複数報告ある場合はまとめて並べて見せる

### Step 7. 実行 → 結果を1行で報告
- 成功/失敗のみ簡潔に。長いログを貼らない

### Step 8. ビルド & デプロイ(GitHub Pages 反映)

DB 更新は GitHub Pages に**自動では反映されない**。push して初めて Actions が走る。
1メッセージ分の連続投入が全部終わったら:

1. (任意)ローカルビルドで確認: `SITE_PASSWORD=test1234 .venv/bin/python scripts/build.py`
2. `git add data/reports.db`
3. `git commit -m "data: <内容を1行で>"`
4. `git push`(Actions が自動デプロイ、数十秒で反映)

> **commit/push は必ずユーザー明示指示があってから実行**(CLAUDE.md §17)。
> Claude 側からは「投入終わりました。commit & push しますか?」と確認するだけ。
> 1日に複数回貼り付けが続く場合は、最後に1回まとめて commit したほうがログがきれい。

## 3. 毎回聞くこと(独断禁止リスト)

| 何を聞くか | 良い聞き方の例 | NG な聞き方 |
|---|---|---|
| 施策の特定 | 「これは TikTok の話、紹介の話、バースデーの話、退店の話、どれですか?」 | 「initiative_id=1 ?」 |
| キャストの特定 | 「これは『えま』さんで合ってますか?」(類似名が複数いるとき) | (なし、これは普通の確認) |
| 進捗の変化 | 「これは『進行中』のまま据え置きですか? それとも完了扱い? あるいは『無理』と言われた感じ?」 | 「status を done に?」 |
| 「無理」扱いの確定 | 「この件は『絶対にやらない』と言われた感じですか? それともまだ説得の余地がある?」(declined への変化は重要) | 「declined にしますか?」 |
| 完了の判定(施策1 TikTok) | 「これは動画もらえてゴール達成扱いにしますか? それともまだ追跡中?」 | 「done にしますか?」 |
| 完了の判定(施策2 紹介) | 「実際に友達/知り合いを連れてきた段階ですか? それともまだ来てくれそう段階?」 | 「done にしますか?」 |
| 完了の判定(施策3 バースデー) | 「日程が確定した感じですか?(『6/15でやります』レベル)」 | 「done にしますか?」 |
| バースデー特殊ルール | 「バースデーは『やらない』と言われても完了/無理扱いにせず、進行中のまま据え置きにします(まだ説得余地ありとみなす運用、CLAUDE.md §6)」と一言添える | declinedで投入 |
| 反応(報告内のニュアンス) | 「キャストさんの反応は良かった/普通/嫌そう のどれが近いですか? 読めなければスキップで」 | 「reaction は positive ?」 |
| 退店の確度 | **聞かない**(本人発言ベースのみ登録、常に `confirmed` 固定運用、2026-05-11確定 / CLAUDE.md §12)。観測/推測の話は DB に入れない | 「やめる温度感は?」 |
| 退店時期 | 「退店時期、◯月◯日でいいですか? それとも未定で登録?」**必ず「未定」の選択肢を含めて聞く**(NULL OK / CLAUDE.md §12)。明示日が書かれているならその確認、書かれていないなら未定で投入 | 「expected_quit_date は?」 / 無理に確定日を埋める |
| 退店理由 | 「理由は『家庭の事情』そのままで登録します。文言変えますか?」 | (これは自然) |
| 報告日 (本文に明記なし) | 「この報告いつのぶんですか? 今日(2026-MM-DD)で登録していいですか?」(基本ユーザーが書いてくれる、忘れていたら必ず聞く) | Claudeが「今日でいいや」と勝手に投入 |
| 日付の解釈 | 「『先週』『昨日』『今日』は何月何日のことですか?」 | (自然) |
| 履歴メモの文言 | 「履歴のメモはこの内容で残します: 『◯◯◯』。これで OK ですか?」 | 「content は『◯◯』で?」 |
| 現状サマリの文言 | 「一覧画面に出る今の状況メモはこれにします: 『◯◯◯』。OK?」 | 「comment は?」 |
| 重複検出ヒット | 「同じ日に既に同じキャストの同じ施策の報告があります。上書きで更新? それとも別の出来事として残す?」 | 黙って --force |
| 巻き戻し(完了/無理→他) | 「これ前に『完了/無理』扱いになってる件です。やっぱり違ったと取り消す感じですか? 取り消すなら強制更新します」 | 普通の更新みたいに見せる |
| commit/push | 「投入終わりました。commit & push しますか?」 | 自動 commit/push |

## 4. 独断で OK なこと(都度確認しなくていい)

- **報告主の黒服を担当黒服から自動引き当て**(--by の自動補完、CLAUDE.md §14)
- `cast_initiative_status` 行がまだ無い場合の自動 upsert
- 元の文章を `--raw` にコピーすること(§5 のルール通り)
- 同 status のときに履歴メモだけ更新すること(updated_at は触らない)
- 投入前の現状確認(`show-status` / `show-cast` / `list-recent-reports`)を Claude 判断で打つこと

## 5. フィールドの埋め方(運用合意済み)

報告 1件ごとに 3つの文言が必要。書き分けはこう:

| 何を入れるか | 説明 | 文字数目安 |
|---|---|---|
| 履歴のメモ(reports.content) | その日の出来事を**短く要約**したログ。一覧画面に直接出る | **50字以内** |
| 元の文章(reports.raw_text) | 報告の**該当部分を改行込みでそのままコピー**。後から「あれ何だっけ」を解決する用 | **基本必須**で省略しない |
| 現状サマリ(cast_initiative_status.comment) | 一覧画面の「今こうなってます」一行。**上書きで最新を保持** | 1行(30-40字) |

> 履歴のメモと現状サマリは同じ文章で良いことが多い。
> 迷ったら同じ文章を両方に入れて OK。

## 6. 退店リスク 1サイクル例(のん/あすか)

ユーザー入力:

```
のん 5月前半

家庭の事情

あすか

時期は未定

痩せたので錦の夜に行きたい
```

Claude の応答(理想形):

> 退店の話として 2件読み取りました。順に確認させてください。
> (※温度感は聞かない:本人発言ベースのみ登録する運用、CLAUDE.md §12)
>
> **1人目: のんさん**
> - 「家庭の事情で 5月前半に辞める」と書いてあります
> - 退店時期、5月前半 → 2026-05-15 で登録していいですか? それとも未定で?
> - 理由はそのまま「家庭の事情」で登録します
>
> **2人目: あすかさん**
> - 「痩せたから錦の夜に行きたい(=他店に移りたい)、時期未定」と書いてあります
> - 退店時期、本文どおり「未定」で登録します(NULL、確定したら後から `update-quit-risk` で入れる)
> - 理由は「痩せたので錦の夜に行きたい」そのままで登録します

ユーザー返答待ち → 確定したら以下のコマンドを提示し、実行可否を最後にもう一度確認:

```bash
.venv/bin/python scripts/ingest.py add-quit-risk \
  --cast のん --certainty confirmed --date 2026-05-15 \
  --reason "家庭の事情" --by 鴇田 \
  --report-date 2026-05-11 \
  --content "5月前半に辞める(家庭の事情)" \
  --raw "のん 5月前半

家庭の事情"

# あすかは時期未定 → --date 省略(NULL)
.venv/bin/python scripts/ingest.py add-quit-risk \
  --cast あすか --certainty confirmed \
  --reason "痩せたので錦の夜に行きたい" --by 鴇田 \
  --report-date 2026-05-11 \
  --content "錦の夜への移籍意向、時期未定" \
  --raw "あすか

時期は未定

痩せたので錦の夜に行きたい"
```

→ 実行 → ユーザーに「commit & push しますか?」と確認

> ※ `--by` は担当黒服を自動引き。あすかは「鴇田/昼」担当(CLAUDE.md §14)。
> ※ `--certainty` は本人発言ベース固定で常に `confirmed`(温度感は聞かない、CLAUDE.md §12)。
> ※ 時期未定なら `--date` を省略(NULL になる、`(未定)` 表示)。

## 7. 特殊フロー / 例外ケース

### 7-1. 同日同キャスト同施策の重複報告

`add-report` は同日に既に同件があると `--force` を要求してエラーになる。

**Claude の動き:**
1. ユーザーに状況を伝え「同じ日に既に同じ報告があります。**新しい内容で上書き**しますか? それとも**別の出来事として2件目を残す**?」と確認
2. 上書き → `--force` 付きで内容更新(report 1件のまま、content/reaction 上書き)
3. 別件として残す → `--force` 付きでもう1件 add-report(report 2件になる)

### 7-2. 完了/無理(done / declined)から別状態への巻き戻し

`TERMINAL_STATUSES = ("done", "declined")` から動かすには `--force` 必須(CLAUDE.md §7)。
**勝手に巻き戻ししない**(CLAUDE.md §18)。

**Claude の動き:**
1. ステータス変化が必要そうな時は事前に `show-status` で現状確認
2. 完了/無理だった → ユーザーに「これ前に『完了/無理』扱いの件です。取り消して進行中に戻しますか? もしくは別の状態に?」確認
3. ユーザー明示 OK → `--force` 付きで状態変更
4. ユーザー躊躇 → 据え置き、コメントだけ更新

### 7-3. バースデー(施策3)で「やらない」と言われた

施策3(バースデーイベント)は **declined を使わない運用**(CLAUDE.md §6, §18)。
「絶対やらない」と言われても **in_progress 据え置き**。コンサルとしてはまだ説得の余地ありとみなす。

**Claude の動き:**
- 報告解釈時点で「バースデーなのでこれは『進行中』のままにします、説得余地ありの運用です(CLAUDE.md §6)」と一言添える
- 「日程確定」のみ done 昇格させる

### 7-4. 1メッセージに複数キャスト × 複数施策

「えまに TikTok と紹介の話両方した」のように 1報告に複数施策が混在 → **施策ごとに reports レコードを分割**(CLAUDE.md §9)。

**Claude の動き:**
- add-report を施策の数だけ繰り返す
- raw_text は当該施策に該当する原文部分を抜粋して入れる(もしくは原文全体を両方に同梱でも可、ユーザーに聞く)
- content は施策ごとに短く要約

### 7-5. 施策4(ひろし顔きき計画)は投入対象外

施策4 は category='standalone'。`add-report` は明示的に拒否する(CLAUDE.md §7)。

**Claude の動き:**
- 「顔きき計画」関連の報告が貼られたら「これは記録対象外の施策(CLAUDE.md §6 / §7)です。投入できません」と返す
- そのまま会話で流す(DB には残さない)

### 7-6. 報告なしで進捗だけ動かしたい時 (`set-status`)

例: 「あれ、TikTok えま、間違って done にしてた、in_progress に戻して」

→ **`set-status` を使う**(reports は作らない、状態だけ動かす)。
※ done/declined 巻き戻しに該当する場合は §7-2 のフロー必須(`--force` + ユーザー確認)。

```bash
.venv/bin/python scripts/ingest.py set-status \
  --cast えま --initiative 1 --status in_progress \
  --comment "前回の done は誤りでした、追跡継続中" --force
```

## 8. キャスト退店処理 / 退店リスクの解消

### 8-1. 実退店した時(`cast quit`)

「のん 5/15に辞めた」報告 → 実退店処理:

```bash
.venv/bin/python scripts/ingest.py cast quit --name のん --date 2026-05-15
```

- `casts.status='quit'` + `casts.quit_date=2026-05-15` になる
- 同時に `quit_risks.is_resolved=1` に自動切り替え(該当行があれば)
- /casts/ ページから「退店済み details」セクションに移る

**Claude の確認**: 「のんさん、5/15で実際に辞められた件で退店処理しますか?」

### 8-2. 辞めない宣言で解消(`update-quit-risk --resolve`)

「のん 家庭の事情解決した、辞めないって」 → 退店リスクから外す:

```bash
.venv/bin/python scripts/ingest.py update-quit-risk --cast のん --resolve
```

- `casts.status='active'` のまま `quit_risks.is_resolved=1` になる
- /quit-risks/ から消えるが、キャスト個別ページの履歴には残る

**Claude の確認**: 「のんさん、辞めないことになったみたいですね。退店リスクからは外しますが、キャストとしては引き続きアクティブで合ってますか?」

### 8-3. 確度の変化(likely → confirmed 等)

「のん やっぱり辞めるって決まったって」 → 確度更新:

```bash
.venv/bin/python scripts/ingest.py update-quit-risk \
  --cast のん --certainty confirmed
```

退店時期も変わったら `--date 2026-05-20` 等を同梱。

### 8-4. 退店時期だけの修正

「のん 5月前半って言ってたけど 5/20 になった」:

```bash
.venv/bin/python scripts/ingest.py update-quit-risk --cast のん --date 2026-05-20
```

### 8-5. 理由の修正

「のん 家庭の事情じゃなくて学業の事情だった」:

```bash
.venv/bin/python scripts/ingest.py update-quit-risk --cast のん --reason "学業の事情"
```

## 10. ユーザーに見せてはいけない / 言い換え対応表

会話で使わない用語と、代わりに使う日本語:

| 内部用語(NG) | ユーザー向け表現(OK) |
|---|---|
| status | 進捗 / 状況 |
| not_started | 未着手 / まだ何もしてない |
| in_progress | 進行中 / 動いてる |
| done | 完了 / ゴール達成 |
| declined | 無理 / やらないと言われた / 断られた |
| reaction (positive/neutral/negative) | 反応(良かった / 普通 / 嫌そう) |
| certainty / confirmed / likely | (運用上は常に「辞めるって本人が言った」= confirmed 固定。ユーザーには温度感を聞かない) |
| expected_quit_date | 退店時期 |
| comment (cast_initiative_status) | 今の状況メモ / 現状の一行 |
| content (reports) | 履歴のメモ / 今日のメモ |
| raw_text | 元の文章 |
| initiative / initiative_id | 施策 / TikTokの話 / 紹介の話 / バースデーの話 / 退店の話 |
| cast / cast_id | キャスト / (名前で呼ぶ) |
| kurofuku / by | 黒服(報告主) / 担当 |
| upsert / FK / UNIQUE / TERMINAL_STATUSES | (出さない) |
| `--force` / `--resolve` 等の CLI フラグ | (出さない、必要なら「強制更新」「解消扱い」と訳す) |

## 13. 本書のメンテ (重要: メタルール)

**本書は「Claude が毎セッション必ず通読する単一情報源」。新しい知見はその場で
ここに追記して次セッションに引き継ぐ。「分かった」だけで放置しない。**

### 必ず追記するトリガー

- **A 群施策(§1-2)の典型フォーマット**を初観測した → 該当節に追記
- **新しい判定パターン**(言い回し → status/reaction)をユーザーと合意した
  → **その場で §1-2-α 早見表に追記**(放置しない)
- 同じパターンを **2回続けてユーザーに聞いた** と気づいた → 早見表 or §3 に追記
- 言い換えの新パターン(技術用語 → 平易な日本語)が出てきた → §10 に追記
- **新しいレイアウト誤読の罠**(改行漏れ / サブヘッダー混入など)に遭遇した → §1-2-β に追記
- 新たな例外フロー(`--force` 必要なケース等)が見つかった → §7 に追記
- 新たな退店パターン(解消 / 確度変化 / 時期修正のクセ等)が見つかった → §8 に追記

### 追記の判断基準

「次セッションの俺が **同じ質問を繰り返すか? / 同じ罠にハマるか?**」が YES なら追記。

### 重要な制約

- **本書の更新内容は必ずユーザー確認を経てから commit/push**
- 自分の解釈だけで追記しない。判断はユーザーと合意してから書く
- 矛盾するパターンを追記する時は、後の判断を残し、矛盾理由を一言添える
